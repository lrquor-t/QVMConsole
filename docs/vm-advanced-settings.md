# 虚拟机高级设置页

## 功能说明

为避免普通用户在创建或编辑虚拟机时误改开发者参数，前端表单新增了独立的“高级设置”页面，并将“启动时冻结 CPU”从原来的“启动与安全”区域迁移到该页面。

首次进入高级设置页时，会先显示一层提示卡片，并将后方正常表单做虚化处理。用户点击“我知道了”后，提示层消失，恢复为普通表单视图，交互效果与开发者监视器的首次提示一致。

RTC 和 SMBIOS 相关设置都在高级设置页中收敛为独立“配置”入口，点击后会弹出表单，统一编辑底层参数，避免主页面展示过多细节字段。

QEMU Guest Agent 也采用同样的“配置入口 + 弹窗表单”方式，主页面仅展示是否启用的摘要，避免在高级设置页直接堆叠过多说明性字段。

对于已创建的虚拟机，高级设置页还额外提供“Domain XML -> 编辑 XML”入口，可直接查看和修改持久化的 libvirt domain XML，用于处理当前面板未暴露的高级参数。该入口仅管理员可见。

## 影响范围

- 创建虚拟机向导：新增“高级设置”步骤
- 编辑虚拟机弹窗：新增“高级设置”标签页
- 原“启动与安全”页面：保留引导顺序、Watchdog、开机自启等常规配置，不再显示“启动时冻结 CPU”
- 高级设置页：默认展示正常表单，但首次进入会弹出风险提示遮罩
- 编辑态高级设置页：新增仅管理员可见的 XML 编辑入口，并在保存时触发二次验证

## 当前包含的选项

### 启动时冻结 CPU

- 默认值：关闭
- 适用场景：调试虚拟机早期启动过程、配合 QEMU Monitor 观察预启动状态
- 启用后行为：虚拟机会以暂停态启动，需要在开发者监视器中执行 `c`，或在面板中再次点击开机恢复运行

### CPU 限制

- 仅管理员可见
- 默认值：无限制
- 输入方式：百分比
- 取值范围：`1-100`，关闭时表示不限速
- 作用：限制虚拟机可使用的 CPU 总时间，按当前配置的 `vCPU` 总能力计算
- 示例：
  - `50%`：限制为当前已分配 CPU 总能力的一半
  - `100%`：允许使用当前已分配 CPU 的全部能力
- 生效方式：
  - 新建 / 导入 / 克隆：创建时直接写入 libvirt `cputune`
  - 编辑已有虚拟机：写入持久配置；运行中的 VM 会尽量同步到运行态
- 注意事项：
  - 当管理员修改 `vCPU` 数量时，系统会自动按原百分比重算 CPU 配额，避免限速比例漂移
  - 该限制是 VM 级 CPU 时间限制，不是单个 vCPU 的独立倍率开关

### APIC

- 默认值：开启
- 作用：控制虚拟机是否向来宾系统暴露 APIC（高级可编程中断控制器）
- 适用场景：
  - 普通 Linux / Windows 虚拟机保持默认开启
  - 仅在排查非常早期的引导兼容性、特殊老系统中断初始化异常时再尝试关闭
- 注意事项：
  - 关闭后可能影响来宾系统对中断控制器的识别与启动过程
  - Windows 虚拟机如果已经启用了较完整的 Hyper-V 兼容特性，通常更不建议随意关闭
- 生效方式：
  - 新建 / 导入 / 克隆：在定义虚拟机 XML 时直接写入
  - 编辑已有虚拟机：写入持久配置，运行中的虚拟机建议关机后修改并重新开机生效

### PAE

- 默认值：开启
- 作用：控制虚拟机是否向来宾系统暴露 PAE（Physical Address Extension，物理地址扩展）
- 适用场景：
  - x86 老系统
  - 32 位系统的大内存兼容场景
  - 启动兼容性排查时，需要显式对比是否启用 PAE
- 注意事项：
  - 64 位系统通常不需要手动调整
  - 非 x86 架构会自动忽略该设置，不会强行写入无效 XML
- 生效方式：
  - 新建 / 导入 / 克隆：在定义虚拟机 XML 时直接写入
  - 编辑已有虚拟机：写入持久配置，运行中的虚拟机建议关机后修改并重新开机生效

### 显示设备

- 可选值：
  - `VirtIO`：高性能，适合 Linux 或已安装 VirtIO 显示驱动的系统
  - `VGA`：兼容模式，适合 Windows 安装阶段或常规兼容性优先场景
  - `VMVGA`：适合 VMware 嵌套 KVM 场景下做兼容性尝试
  - `Cirrus`：更保守的排障模式，仅建议在显示异常时临时尝试
- 默认值：
  - Linux / 其他系统：`VirtIO`
  - Windows：`VGA`
- 适用场景：
  - Windows 安装阶段 VNC 明显卡顿
  - VMware/ESXi 嵌套环境中的 KVM 虚拟机图形延迟偏高
  - 需要在兼容性和性能之间手动权衡
- 建议：
  - 物理宿主或 Linux 系统优先使用 `VirtIO`
  - Windows 若使用 VNC 安装系统，优先尝试 `VGA`
  - 如果宿主是 VMware 嵌套环境，可继续尝试 `VMVGA`
  - 即使切换后改善有限，Windows 装机完成后仍建议改用 `RDP`
- 生效方式：
  - 新建 / 导入 / 克隆：在定义虚拟机 XML 时直接写入
  - 编辑已有虚拟机：需要在关机状态下修改并重新开机生效

### RTC 时间基准

- 可选值：`UTC` / `本地时间`
- 默认值：
  - Linux / 其他系统：`UTC`
  - Windows：`本地时间`
- 适用场景：控制虚拟机硬件时钟 RTC 使用 UTC 还是宿主机本地时间
- 建议：
  - Linux 通常保持 `UTC`
  - Windows 通常保持 `本地时间`
- 生效方式：
  - 创建虚拟机时立即写入定义
  - 编辑已有虚拟机时写入持久配置，运行中的虚拟机需要重启后生效

### RTC 开始日期

- 默认值：`now`
- 作用：控制虚拟机启动时 RTC 的初始时间
- 推荐用法：
  - 普通情况下保持 `now`
  - 仅在测试固定时间启动、排查与时间相关的问题时再填写具体时间
- 支持格式：
  - `now`
  - Unix 时间戳
  - `RFC3339`
  - `YYYY-MM-DD HH:mm:ss`
- 注意：
  - 当保持 `now` 时，继续使用上方的 RTC 时间基准（UTC / 本地时间）
- 当填写固定时间后，会切换为固定起始时间模式，运行中的虚拟机需要重启后生效

### QEMU Guest Agent

- 默认值：关闭
- 作用：
  - 让宿主机读取来宾系统内部信息
  - 提供更可靠的来宾关机协作
  - 为冻结文件系统后再执行快照等场景提供通道能力
- 使用建议：
  - Linux / Windows 服务器建议启用
  - 虚拟机内部必须已安装 `qemu-guest-agent`
  - 如果只是普通桌面或临时测试机，可保持默认关闭
- 当前面板保留字段：
  - `使用 QEMU Guest Agent`
- 当前面板未暴露字段：
  - `guest-trim`
  - `agent type`
  - `freeze-fs-on-backup`
- 设计说明：
  - 当前面板基于 libvirt 维护虚拟机定义，启用时会向 domain XML 写入标准 QEMU Guest Agent 通道
  - 暂未引入仅对特定备份链路生效的额外策略字段，避免让高级页出现当前面板尚未使用的配置
- 生效方式：
  - 新建 / 导入 / 克隆：创建定义时直接写入
  - 编辑已有虚拟机：写入持久配置，建议关机后修改并重新开机生效

### SMBIOS 设置（类型 1）

- 可选字段：
  - `base64`
  - `family`
  - `manufacturer`
  - `product`
  - `serial`
  - `sku`
  - `uuid`
  - `version`
- 作用：向来宾系统暴露一组虚拟机器身份信息，例如厂商、产品、序列号、UUID
- 适用场景：
  - 软件授权绑定
  - 资产识别
  - 迁移兼容或模拟特定硬件身份
- 建议：
  - 普通业务虚拟机一般不要修改
  - 仅在明确知道来宾系统依赖这些字段时再设置
- `base64` 说明：
  - 仅作为写入辅助开关
  - 启用后，面板会先将填写值按 Base64 解码，再写入 libvirt XML
  - 详情回显始终显示解码后的实际值
- `uuid` 说明：
  - 新建、导入、克隆时可填写标准 UUID
  - libvirt 要求 SMBIOS UUID 与虚拟机顶层 UUID 保持一致
  - 为避免直接改写已存在虚拟机身份，编辑页仅支持查看当前 UUID，不支持直接修改
- 生效方式：
  - 新建 / 导入 / 克隆：在定义虚拟机 XML 时写入
  - 编辑已有虚拟机：写入持久配置，运行中的虚拟机需重启后生效

### Domain XML 编辑

- 仅管理员在编辑已有虚拟机时显示
- 作用：直接查看并编辑当前虚拟机的持久化 libvirt domain XML
- 设计边界：
  - 读取 `virsh dumpxml --inactive`
  - 保存调用 `virsh define`
  - 不支持通过该入口修改虚拟机名称
- 交互说明：
  - 保存时会触发二次验证
  - 保存完成后会刷新当前编辑表单和磁盘列表
  - 若普通表单还有未提交改动，保存 XML 后这些未提交改动会被刷新覆盖
- 使用建议：
  - 优先在关机状态下修改
  - 仅在当前表单未覆盖到该配置项时再使用
  - 不熟悉 libvirt XML 时不要直接修改

## 使用建议

- 普通业务虚拟机建议保持默认关闭
- 仅在排查启动异常、调试驱动或验证底层引导流程时启用
- 如果不熟悉 QEMU Monitor，建议不要修改该页面配置

### CPU 拓扑模式

- 可选值：
  - `auto`（自动）：Windows 来宾自动使用单插槽多核心，Linux 来宾保持 libvirt 默认行为
  - `single_socket`（单插槽多核心）：将所有 vCPU 暴露为单插槽 `sockets=1, dies=1, threads=1`，cores 等于 vCPU 数量
  - `host_default`（宿主默认拓扑）：移除显式拓扑，由 libvirt/QEMU 根据 vCPU 数量自动分配
- 默认值：`auto`
- 适用场景：
  - Windows 10/11 建议使用 `auto` 或 `single_socket`，避免将多核识别为多颗物理 CPU
  - Linux 来宾通常保持 `auto` 即可
- 生效方式：
  - 新建 / 导入 / 克隆：创建定义时直接写入
  - 编辑已有虚拟机：需要在关机状态下修改并重新开机生效

### CPU 拓扑与 vCPU 数量同步

- libvirt 要求 domain XML 中的 `vcpu` 值必须等于 `sockets × dies × cores × threads`
- 编辑虚拟机 vCPU 数量时，系统会自动同步 CPU 拓扑以保证等式成立：
  - 如果 XML 中不存在显式拓扑，使用 `virsh setvcpus` 命令设置
  - 如果 XML 中存在显式拓扑，同时修改 `vcpu` 和 `<topology>` 后通过 `virsh define` 更新持久化配置
- 同时检查持久化和在线配置中的拓扑信息，避免因配置不一致导致设置失败
- 如果当前拓扑参数无法整除目标 vCPU 数量，会返回明确的错误提示

### CPU 热添加

- 启用后，系统会将虚拟机的最大 vCPU 设置为宿主机 CPU 核心总数，后续可在不超过该上限的范围内随时热添加 vCPU，无需重启
- 启用时域 XML 使用 `<vcpu placement='static' current='N'>M</vcpu>` 格式（N=当前，M=最大），CPU 拓扑会按最大值 M 配置以确保一致性
- 关闭时不设置热添加上限，最大 vCPU 等于当前 vCPU
- 新建虚拟机启用热添加后，需重启一次后热添加功能生效（因为 libvirt 不支持在线修改 vCPU 最大值）
- 编辑已有虚拟机时切换热添加状态，会在持久化配置中写入对应的 max_vcpu，重启后生效
